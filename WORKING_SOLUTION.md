# Платформа распределённых волонтёрских вычислений

## White Paper

---

## 1. Проблема

Современная наука генерирует вычислительные задачи, которые не помещаются на одну машину: перебор параметров по миллионам комбинаций, обработка десятков тысяч геномных образцов, симуляции Монте-Карло с миллиардами итераций. У учёного сегодня три варианта:

| Вариант | Проблема |
|---|---|
| Облако (AWS, GCP, Azure) | Дорого. Требует знания инфраструктуры. |
| Университетский HPC-кластер | Очереди, бюрократия, квоты, ограниченный доступ. |
| BOINC | Учёный должен поднять сервер, скомпилировать приложение под каждую ОС, привлечь волонтёров, заниматься PR. Проекты умирают не из-за плохой науки, а из-за операционной нагрузки. |

Создатель BOINC описывает корневую проблему: платформа — это **фреймворк для создания проектов распределённых вычислений**, а не **сервис**. Каждая исследовательская группа должна самостоятельно развернуть серверы, разработать приложения, скомпилировать их под несколько платформ и набрать волонтёров. Это "severely stunted the growth of BOINC" (серьёзно затормозило рост BOINC).

**Наш тезис:** барьер — не технология, а операционная нагрузка. Убери её — и волонтёрские вычисления станут доступны любому учёному, у которого есть скрипт и датасет.

---

## 2. Решение в одном предложении

> Цикл for, но на 10 000 чужих компьютерах.

Платформа берёт скрипт и датасет, разбивает работу на тысячи волонтёрских машин и возвращает собранный результат. Учёный не управляет инфраструктурой, ничего не компилирует и не набирает волонтёров.

**Мы конкурируем с облаком и очередями HPC, а не с локальным запуском скрипта.** Платформа полезна, когда учёному нужно прогнать одно и то же вычисление на тысячах входов — задача, которая заняла бы часы или дни на одной машине.

## 3. Пользовательский опыт

### 3.1. Путь учёного

Весь процесс — три шага на одной веб-странице.

**Шаг 1. Загрузить данные.**

Учёный загружает CSV-файл. Платформа отображает его как привычную таблицу:

```
| gene_id | sequence       | organism    |
|---------|----------------|-------------|
| BRCA1   | ATGCGATC...    | H. sapiens  |
| TP53    | GCTTAACG...    | M. musculus |
| MYC     | TTGAACCG...    | R. norveg.  |
| ... ещё 49 997 строк                   |
```

**Шаг 2. Загрузить скрипт.**

Учёный загружает свой скрипт обработки. Платформа мгновенно запускает его на первых 3 строках (dry run) и показывает результат рядом с исходными данными:

```
| gene_id | sequence     | organism    |  →  gc_content  |  →  length |
|---------|--------------|-------------|-----------------|------------|
| BRCA1   | ATGCGATC...  | H. sapiens  |  →  0.58        |  →  1842   |
| TP53    | GCTTAACG...  | M. musculus |  →  0.51        |  →  2109   |
| MYC     | TTGAACCG...  | R. norveg.  |  →  0.61        |  →  987    |
```

Учёный видит свои данные, видит что скрипт с ними делает, и убеждается в корректности результата — всё это до начала какого-либо распределённого вычисления.

**Шаг 3. Одна кнопка.**

Под превью:

> Обработано 3 из 50 000 строк. Одна строка ≈ 0.2 сек.
> Полный запуск локально: ~2.8 часа. На платформе: **~1 минута.**
>
> **[ Обработать все ]**

Учёный нажимает кнопку. Платформа разбивает датасет, распределяет чанки по доступным воркерам, собирает результаты и формирует единый файл для скачивания.

Без настройки проекта. Без развёртывания серверов. Без набора волонтёров. Без компиляции.

### 3.2. Форматы входных данных

Платформа принимает три типа входных данных, определяемых автоматически по расширению файла:

| Загрузка | Платформа интерпретирует как | Стратегия разбиения |
|---|---|---|
| `.csv` / `.tsv` | Табличные данные | Каждая строка (или группа строк) = одна задача |
| `.zip` / `.tar.gz` | Набор файлов | Каждый файл = одна задача |
| Только скрипт + числовые поля в веб-форме | Параметрический перебор | Каждое значение (или поддиапазон) = одна задача |

Никакого файла манифеста. Формат входных данных **и есть** манифест.

JSONL (`.jsonl`) поддерживается как дополнительный четвёртый формат — каждая строка является независимым JSON-объектом.

### 3.3. Контракт скрипта

Контракт между скриптом учёного и платформой — два предложения:

1. **Читай вход из stdin.**
2. **Пиши результат в stdout.**

Это стандартная Unix-конвенция пайпов, поддерживаемая каждым языком программирования. Полностью рабочий скрипт:

```python
import sys

for line in sys.stdin:
    gene = line.strip()
    print(calc_gc(gene))
```

```r
input <- readLines("stdin")
for (line in input) {
    cat(calc_gc(line), "\n")
}
```

Для параметрического перебора значения приходят через аргументы командной строки (`sys.argv`).

#### Автоматическая адаптация скрипта

Многие учёные пишут скрипты, которые читают файлы напрямую, а не из stdin:

```python
df = pd.read_csv("data.csv")
```

Когда платформа обнаруживает такой паттерн во время dry run, она предлагает исправление в один клик:

> Ваш скрипт читает `data.csv` напрямую. Чтобы разбить данные между воркерами, нужна одна правка:
>
> **Заменить:**
> ```python
> df = pd.read_csv("data.csv")
> ```
> **На:**
> ```python
> import sys
> df = pd.read_csv(sys.stdin)
> ```
> **[ Применить ]** **[ Редактировать вручную ]**

Учёный видит конкретное изменение, понимает его и применяет одним кликом.

### 3.4. Обнаружение агрегации

Некоторые скрипты выдают один результат на все входные данные (среднее, сумма, группировка), а не один результат на строку. Если платформа разобьёт такой скрипт по воркерам, каждый чанк выдаст свой агрегат — некорректный результат.

Обнаружение автоматическое: при dry run на 3 строках скрипт выдаёт 1 строку результата вместо 3. Платформа спрашивает:

> Ваш скрипт выдаёт один результат на все входные строки.
> Это агрегация (среднее, итог, сводка)? **[ Да ]** **[ Нет, это баг ]**

Если да — платформа либо запускает скрипт на одной машине без разбиения, либо применяет двухфазную стратегию: map-фаза на воркерах, затем reduce-фаза на сервере.

### 3.5. Путь волонтёра

**Браузерный режим (zero-install):**

Волонтёр открывает веб-страницу. WebAssembly-воркер запускается во вкладке браузера. Пока вкладка открыта, CPU волонтёра обрабатывает научные задачи. Закрытие вкладки прекращает участие. Без регистрации, без установки, без настройки.

Страница показывает в реальном времени:
- Какому исследованию помогает волонтёр
- Сколько задач обработано
- Оценку вклада (например: "Вы обработали 342 задачи для геномного исследования MIT")

**Режим нативного агента:**

Для преданных волонтёров лёгкий фоновый демон предоставляет больше ресурсов: доступ к GPU, больше RAM, работа при закрытом браузере. Это режим для опытных пользователей, аналогичный традиционному BOINC, но без привязки к конкретным проектам.

### 3.6. Встраиваемый виджет

Веб-сайты могут встроить виджет "Помоги науке" как альтернативу рекламе — с явного согласия пользователя. Посетитель соглашается, и его CPU помогает научным вычислениям, пока он просматривает сайт. Концептуально похоже на Coinhive (сервис 2017-2019, который использовал CPU посетителей для майнинга Monero), но для науки вместо криптовалюты и с обязательным согласием пользователя.

## 4. Техническая архитектура

### 4.1. Обзор

```
┌──────────────┐        ┌──────────────────┐        ┌──────────────┐
│   Учёный     │───────▶│   Оркестратор     │───────▶│  Волонтёр    │
│  (Веб-UI)    │        │                  │        │  (Браузер)   │
│              │◀───────│  - Очередь задач  │◀───────│              │
│  Загрузка    │        │  - Сплиттер       │        │  WASM-воркер │
│  Превью      │        │  - Агрегатор      │        │  в Web       │
│  Скачивание  │        │  - Dry runner     │        │  Worker      │
└──────────────┘        │  - Верификатор    │        └──────────────┘
                        │                  │
                        │                  │        ┌──────────────┐
                        │                  │───────▶│  Волонтёр    │
                        │                  │◀───────│  (Нативный)  │
                        └──────────────────┘        │              │
                                                    │ WASM-рантайм │
                                                    │ или Docker   │
                                                    └──────────────┘
```

### 4.2. WebAssembly как основной рантайм

WASM — ключевая технология, которая снимает главные технические боли BOINC:

| Проблема BOINC | Решение WASM |
|---|---|
| Нужна компиляция под каждую ОС/архитектуру | Один WASM-бинарник работает везде |
| Волонтёры запускают недоверенный код | WASM-песочница — нет доступа к ФС и сети |
| Нужна VM для изоляции (тяжёлая, медленная) | Песочница WASM почти без накладных расходов |
| Сложные мультиплатформенные системы сборки | Единственная цель компиляции |

Для Python-скриптов платформа использует **Pyodide** (CPython, скомпилированный в WASM). Для R-скриптов — **WebR** (R, скомпилированный в WASM). Учёный пишет обычный Python/R; платформа обеспечивает компиляцию в WASM прозрачно.

### 4.3. Управление зависимостями

Научные скрипты требуют библиотек. Платформа решает это двумя способами:

**Толстый рантайм с предустановленными пакетами.**

Pyodide поставляется с ~200 предкомпилированными WASM-пакетами, включая научное ядро: numpy, scipy, pandas, scikit-learn, matplotlib, networkx, sympy, statsmodels. WebR покрывает ~10 500 рабочих пакетов CRAN, включая полный tidyverse.

Для 80% научных скриптов на Python/R всё уже доступно из коробки.

**Docker-fallback.**

Для случаев, требующих системных библиотек, нестандартных C-расширений или доступа к GPU, учёный указывает ссылку на Docker-образ. Это работает только на нативных агентах, не в браузерах.

### 4.4. Двухуровневая архитектура волонтёров

**Браузерные воркеры (уровень 1):**
- WASM-выполнение внутри Web Worker
- Без установки, без настройки
- Ограничения: только CPU, лимит RAM (~256 МБ линейной памяти WASM), активны только при открытой вкладке
- Максимальный охват: любой человек с браузером может участвовать

**Нативные агенты (уровень 2):**
- Лёгкий демон на Wasmtime/Wasmer (или Docker для сложных задач)
- Доступ к GPU, больше RAM, фоновая работа
- Меньший пул, но более мощный
- Необходим для: задач с большой памятью, GPU-вычислений, Docker-задач, обработки чувствительных данных

Платформа автоматически маршрутизирует задачи на подходящий уровень на основе требований к ресурсам, определённых во время dry run.

## 5. Валидация и безопасность

### 5.1. Три уровня защиты

**Уровень 1: Dry run на сервере.**

Перед распределением работы платформа запускает скрипт на 1 чанке в собственной песочнице. Это ловит:

- Синтаксические ошибки (показываются учёному с подсветкой строки)
- Ошибки выполнения (отсутствующие пакеты, ошибки типов)
- Бесконечные циклы (убиваются по таймауту)
- Чрезмерное потребление ресурсов (слишком медленно для браузерных воркеров)
- Отсутствие вывода (скрипт работает, но ничего не выдаёт)

Dry run — это не скрытый шаг валидации, это **и есть** то превью, которое учёный уже видит. Один механизм, четыре функции: UX, валидация, оценка времени и обнаружение злоупотреблений.

**Уровень 2: Жёсткие лимиты на воркерах.**

| Ресурс | Браузерный воркер | Нативный воркер |
|---|---|---|
| Время на чанк | 30 секунд | 5 минут |
| Память | 256 МБ | 2 ГБ |
| Размер вывода | 10 МБ | 100 МБ |
| Доступ к сети | Нет | Нет |
| Доступ к ФС | Только песочница | Только песочница |

WASM-рантаймы поддерживают **fuel metering** (учёт топлива): в WASM-модуль инжектируются счётчики инструкций, и выполнение останавливается, когда бюджет исчерпан. Это точнее, чем таймаут по времени, потому что измеряет само вычисление, а не прошедшее время — 30 секунд на быстром десктопе и 30 секунд на медленном ноутбуке волонтёра представляют разный объём работы, а fuel metering единообразен.

```rust
// Fuel metering в Wasmtime
let mut store = Store::new(&engine, ());
store.set_fuel(1_000_000_000);
// Выполнение автоматически останавливается при fuel = 0
```

**Уровень 3: Поведенческий анализ.**

Постфактум-эвристики для обнаружения паттернов злоупотребления:

| Сигнал | Интерпретация |
|---|---|
| Высокий CPU + пустой/минимальный stdout | Подозрительно (майнинг, бесконечный цикл) |
| Размер вывода не пропорционален входу | Скрипт игнорирует вход |
| Одинаковый вывод на разных входах | Скрипт игнорирует вход |
| Время выполнения сильно скачет между чанками | Нестабильный скрипт или условное злоупотребление |

### 5.2. Почему крипто-майнинг — не главная угроза

- WASM-майнинг **значительно медленнее** нативного — экономически иррационален
- Майнерам нужен **доступ к сети** для отправки хэшей в пулы — WASM-песочница блокирует это
- Вывод скрипта должен **коррелировать с входом** — майнер не производит осмысленного вывода относительно входа
- Fuel metering ограничивает общий объём вычислений, независимо от того, что скрипт делает

### 5.3. Верификация результатов

Используется проверенная модель BOINC: **избыточное вычисление**. Каждый чанк отправляется нескольким воркерам независимо. Если результаты совпадают (с учётом допустимой погрешности для float-вычислений) — результат принимается. Если расходятся — чанк пересчитывается.

Это простой, надёжный и хорошо изученный подход, работавший в BOINC более 20 лет. Оптимизации (репутационная система, выборочная проверка) возможны в будущем, но на старте избыточность — достаточное решение.

### 5.4. Защита данных

Используется та же модель, что и в BOINC: **данные отправляются на волонтёрские машины в открытом виде**. Если машина обрабатывает данные — она их видит. Это фундаментальное ограничение, а не недостаток платформы.

Для научных данных это, как правило, приемлемо — большинство датасетов (геномные базы, климатические данные, астрономические обзоры) и так публичны. BOINC успешно работает с этой моделью более 20 лет.

Платформа честно об этом сообщает:

> Волонтёры видят фрагменты ваших данных. Для большинства научных задач данные и так публичны. Для чувствительных данных платформа не подходит — обрабатывайте локально или на защищённых кластерах.

## 6. Конкурентный ландшафт

| Решение | Стоимость | Сложность настройки | Параллелизм | Ограничение |
|---|---|---|---|---|
| **Локальная машина** | Бесплатно | Нулевая | 1 машина | Часы/дни на больших задачах |
| **Облако (AWS/GCP)** | $$$ | Высокая (знание инфраструктуры) | Неограниченный | Упирается в бюджет |
| **Университетский HPC** | Бесплатно | Средняя (очереди, квоты) | Высокий | Доступность, бюрократия |
| **BOINC** | Бесплатно | Очень высокая (свой сервер, приложение, PR) | Высокий | Операционная нагрузка убивает adoption |
| **Google Colab** | Бесплатно | Низкая | 1 машина | Лимиты времени, один GPU |
| **Наша платформа** | Бесплатно | **Нулевая** | Высокий | Только позорно параллельные задачи |

Уникальная позиция платформы: **ноль затрат + ноль настройки + высокий параллелизм.** Компромисс в том, что она работает только для embarrassingly parallel нагрузок (независимые задачи, не требующие коммуникации между собой). Это покрывает подавляющее большинство научных пакетных вычислений.

## 7. Риски и честные ограничения

### Проблема холодного старта
"Результат за минуты" требует тысяч активных воркеров. На старте пул волонтёров мал. **Митигация:** начать с облачного бэкенда, который обрабатывает избыток, постепенно перекладывая нагрузку на волонтёров по мере роста пула. Учёные всегда получают результаты; источник вычислений прозрачен, но не является их заботой.

### Проблема курицы и яйца
Учёные не придут без воркеров. Воркеры не придут без видимого научного эффекта. **Митигация:** запуск с одним конкретным сообществом (например, биоинформатики, которым нужно прогонять BLAST-подобные задачи на множестве последовательностей), доказать ценность, затем расширяться. Встраиваемый виджет также раскачивает пул волонтёров независимо от прямого привлечения.

### Не все задачи подходят
Задачи, требующие межузловой коммуникации (MPI-стиль), огромного объёма RAM или специализированного оборудования, не будут работать в браузерном режиме. **Митигация:** быть явным в том, для чего платформа предназначена. "Embarrassingly parallel" — это не ограничение, которое нужно скрывать, а чёткий, честный скоуп, покрывающий большинство научных пакетных вычислений.

### Чувствительность данных
Некоторые области (медицина, социология) работают с регулируемыми данными, которые нельзя отправлять на недоверенные машины. **Митигация:** уровни доверия с верифицированными институциональными воркерами плюс честное сообщение о том, что платформа подходит не для всех типов данных.

## 8. Минимально жизнеспособный продукт (MVP)

MVP проверяет одну гипотезу: **будут ли учёные использовать платформу, если барьер действительно нулевой?**

### Скоуп

- **Веб-портал**: загрузка `.csv` + `.py` скрипта, просмотр превью, нажатие "Обработать все", скачивание результата
- **Один рантайм**: Python через Pyodide (numpy, scipy, pandas предустановлены)
- **Только браузерные воркеры**: WASM в Web Workers, без нативного агента
- **Один режим разбиения**: строки CSV
- **Dry run + превью**: ключевой UX-дифференциатор
- **Базовый fuel metering**: лимиты по времени и инструкциям

### За пределами MVP

- Нативный агент, Docker-fallback
- Поддержка R / WebR
- ZIP/файловый вход
- Режим параметрического перебора
- Уровни доверия и контроль чувствительности данных
- Встраиваемый виджет
- Репутационная система (на старте используем избыточное вычисление для верификации)

### Критерии успеха

- Учёный без знания платформы загружает скрипт и датасет и получает корректные результаты быстрее, чем при локальном запуске
- Весь процесс (загрузка → превью → обработка → скачивание) занимает менее 5 минут пользовательского времени
- Волонтёры в браузере могут участвовать, просто открыв веб-страницу
